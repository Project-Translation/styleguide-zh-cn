Google HTML/CSS 风格指南

背景

本文档定义了 HTML 和 CSS 的格式和风格规则。旨在提高协作、代码质量，并支持基础设施。它适用于使用 HTML 和 CSS 的原始工作文件，包括 Sass 和 GSS 文件。只要保持总体代码质量，工具可以自由地进行混淆、压缩和编译。

通用

通用风格规则

协议

尽可能对嵌入资源使用 HTTPS。

对于图像和其他媒体文件、样式表和脚本，始终使用 HTTPS (https:)，除非相应文件无法通过 HTTPS 访问。

<!-- 不推荐：省略了协议 -->
<script src="//ajax.googleapis.com/ajax/libs/jquery/3.4.0/jquery.min.js"></script>

<!-- 不推荐：使用 HTTP -->
<script src="http://ajax.googleapis.com/ajax/libs/jquery/3.4.0/jquery.min.js"></script>

<!-- 推荐 -->
<script src="https://ajax.googleapis.com/ajax/libs/jquery/3.4.0/jquery.min.js"></script>

/* 不推荐：省略了协议 */
@import '//fonts.googleapis.com/css?family=Open+Sans';

/* 不推荐：使用 HTTP */
@import 'http://fonts.googleapis.com/css?family=Open+Sans';

/* 推荐 */
@import 'https://fonts.googleapis.com/css?family=Open+Sans';

通用格式规则

缩进

每次缩进 2 个空格。

不要使用制表符或混合使用制表符和空格进行缩进。

<ul>
  <li>Fantastic
  <li>Great
</ul>

.example {
  color: blue;
}

大小写

仅使用小写字母。

所有代码必须使用小写字母：这适用于 HTML 元素名称、属性、属性值（除非是 text/CDATA）、CSS 选择器、属性和属性值（字符串除外）。

<!-- 不推荐 -->
<A HREF="/">Home</A>

<!-- 推荐 -->
<img src="google.png" alt="Google">

/* 不推荐 */
color: #E5E5E5;

/* 推荐 */
color: #e5e5e5;

尾随空白

删除尾随空白字符。

尾随空白字符是不必要的，可能会使差异比较复杂化。

<!-- 不推荐 -->
<p>What?_

<!-- 推荐 -->
<p>Yes please.

通用元规则

编码

使用 UTF-8（无 BOM）。

确保您的编辑器使用 UTF-8 作为字符编码，且不带字节顺序标记。

在 HTML 模板和文档中通过 <meta charset="utf-8"> 指定编码。不要为样式表指定编码，因为它们假定使用 UTF-8。

（关于编码以及何时和如何指定它们，可以在 HTML 和 CSS 中处理字符编码 中找到更多信息。）

注释

根据需要解释代码，尽可能做到这一点。

使用注释来解释代码：它涵盖了什么，它的目的是什么，为什么使用或偏好相应的解决方案？

（此项是可选的，因为要求代码始终完全文档化并不现实。HTML 和 CSS 代码的里程数可能会因项目的复杂性而大不相同。）

行动项目

使用 TODO 标记待办事项和行动项目。

通过仅使用关键字 TODO 来突出显示待办事项，而不是使用其他常见格式如 @@。

在冒号后附加行动项目，如 TODO: 行动项目。

{# TODO: 重新审视居中。 #}
<center>Test</center>

<!-- TODO: 删除可选标签。 -->
<ul>
  <li>Apples</li>
  <li>Oranges</li>
</ul>

HTML

HTML 样式规则

文档类型

使用 <!doctype html>。

始终在文档开头包含 <!doctype html>，以确保您的 HTML 处于 无怪癖模式。

没有文档类型的文档将以“怪癖模式”渲染，而使用不同文档类型的文档可能会以“有限怪癖模式”渲染。这些模式不遵循各种核心 HTML 和 CSS 结构的广泛理解和广泛记录的行为，可能会导致细微的失败和不兼容性，特别是在重用期望无怪癖模式的代码时。

HTML 有效性

尽可能使用有效的 HTML。

除非由于文件大小方面的性能目标无法实现，否则使用有效的 HTML 代码。

使用如 W3C HTML 验证器 这样的工具进行测试。

使用有效的 HTML 是一个可衡量的基线质量属性，有助于了解技术要求和限制，并确保正确使用 HTML。

<!-- 不推荐 -->
<title>Test</title>
<article>This is only a test.

<!-- 推荐 -->
<!doctype html>
<meta charset="utf-8">
<title>Test</title>
<article>This is only a test.</article>

语义

根据其用途使用 HTML。

使用元素（有时错误地称为“标签”）来实现它们被创建的目的。例如，使用标题元素来表示标题，使用 p 元素来表示段落，使用 a 元素来表示锚点等。

根据其用途使用 HTML 对可访问性、重用性和代码效率至关重要。

<!-- 不推荐 -->
<div onclick="goToRecommendations();">All recommendations</div>

<!-- 推荐 -->
<a href="recommendations/">All recommendations</a>

多媒体备选内容

为多媒体提供替代内容。

对于多媒体，如图像、视频、通过 canvas 实现的动画对象，确保提供替代访问方式。对于图像，这意味着使用有意义的替代文本（alt），对于视频和音频，如果可用，则提供字幕和脚本。

提供替代内容对可访问性至关重要：盲人用户在没有 @alt 的情况下几乎无法判断图像的内容，其他用户也可能无法理解视频或音频的内容是什么。

（对于那些 alt 属性会引入冗余的图像，以及对于那些目的纯粹是装饰性的且您无法立即使用 CSS 的图像，使用无替代文本，如 alt=""。）

<!-- 不推荐 -->
<img src="spreadsheet.png">

<!-- 推荐 -->
<img src="spreadsheet.png" alt="Spreadsheet screenshot.">

关注点分离

将结构与表现与行为分离。

严格区分结构（标记）、表现（样式）和行为（脚本），并尽量将三者之间的交互保持在绝对最小限度。

也就是说，确保文档和模板仅包含 HTML，并且这些 HTML 仅用于结构目的。将所有表现内容移到样式表中，将所有行为内容移到脚本中。

此外，通过从文档和模板中链接尽可能少的样式表和脚本，尽量减少接触面积。

将结构与表现与行为分离对维护至关重要。更改 HTML 文档和模板总是比更新样式表和脚本更昂贵。

<!-- 不推荐 -->
<!doctype html>
<title>HTML sucks</title>
<link rel="stylesheet" href="base.css" media="screen">
<link rel="stylesheet" href="grid.css" media="screen">
<link rel="stylesheet" href="print.css" media="print">
<h1 style="font-size: 1em;">HTML sucks</h1>
<p>I’ve read about this on a few sites but now I’m sure:
  <u>HTML is stupid!!1</u>
<center>I can’t believe there’s no way to control the styling of
  my website without doing everything all over again!</center>

<!-- 推荐 -->
<!doctype html>
<title>My first CSS-only redesign</title>
<link rel="stylesheet" href="default.css">
<h1>My first CSS-only redesign</h1>
<p>I’ve read about this on a few sites but today I’m actually
  doing it: separating concerns and avoiding anything in the HTML of
  my website that is presentational.
<p>It’s awesome!

实体引用

不要使用实体引用。

假设文件和编辑器以及团队之间使用相同的编码（UTF-8），则无需使用像 &mdash;、&rdquo; 或 &#x263a; 这样的实体引用。

唯一的例外适用于在 HTML 中具有特殊含义的字符（如 < 和 &）以及控制或“不可见”字符（如不间断空格）。

<!-- 不推荐 -->
The currency symbol for the Euro is &ldquo;&eur;&rdquo;.

<!-- 推荐 -->
The currency symbol for the Euro is “€”.

可选标签

省略可选标签（可选）。

为了优化文件大小和提高可扫描性，请考虑省略可选标签。HTML5规范定义了哪些标签可以省略。

（这种方法可能需要一段时间来确立为更广泛的指导方针，因为它与通常教给网页开发者的内容大不相同。为了保持一致性和简洁性，最好省略所有可选标签，而不仅仅是选择性地省略。）

<!-- 不推荐 -->
<!doctype html>
<html>
  <head>
    <title>Spending money, spending bytes</title>
  </head>
  <body>
    <p>Sic.</p>
  </body>
</html>

<!-- 推荐 -->
<!doctype html>
<title>Saving money, saving bytes</title>
<p>Qed.

type 属性

省略样式表和脚本的 type 属性。

除非不使用 CSS，否则不要在样式表中使用 type 属性；除非不使用 JavaScript，否则不要在脚本中使用 type 属性。

在这些上下文中指定 type 属性是不必要的，因为 HTML5 暗示默认值为 text/css 和 text/javascript。即使对于旧版浏览器，这也是可以安全执行的。

<!-- 不推荐 -->
<link rel="stylesheet" href="https://www.google.com/css/maia.css"
    type="text/css">

<!-- 推荐 -->
<link rel="stylesheet" href="https://www.google.com/css/maia.css">

<!-- 不推荐 -->
<script src="https://www.google.com/js/gweb/analytics/autotrack.js"
    type="text/javascript"></script>

<!-- 推荐 -->
<script src="https://www.google.com/js/gweb/analytics/autotrack.js"></script>

id 属性

避免使用不必要的 id 属性。

对于样式使用 class 属性，对于脚本使用 data 属性更合适。

在绝对需要 id 属性时，始终在值中包含连字符，以确保它不匹配 JavaScript 标识符语法，例如使用 user-profile 而不是仅使用 profile 或 userProfile。

当一个元素具有 id 属性时，浏览器会将其作为全局 window 原型上的命名属性提供，这可能会导致意外行为。虽然包含连字符的 id 属性值仍然可以作为属性名使用，但这些不能作为全局 JavaScript 变量引用。

<!-- 不推荐：`window.userProfile` 将解析为引用 <div> 节点 -->
<div id="userProfile"></div>

<!-- 推荐：需要 `id` 属性且其值包含连字符 -->
<div aria-describedby="user-profile">
  …
  <div id="user-profile"></div>
  …
</div>

HTML 格式规则

通用格式

对每个块、列表或表格元素使用新行，并缩进每个此类子元素。

不论元素的样式如何（因为 CSS 允许元素根据 display 属性承担不同的角色），将每个块、列表或表格元素放在新行上。

此外，如果它们是块、列表或表格元素的子元素，则应缩进它们。

（如果遇到列表项之间的空白问题，可以将所有 li 元素放在一行上。建议代码检查工具对此发出警告而不是错误。）

<blockquote>
  <p><em>Space</em>, the final frontier.</p>
</blockquote>

<ul>
  <li>Moe
  <li>Larry
  <li>Curly
</ul>

<table>
  <thead>
    <tr>
      <th scope="col">Income
      <th scope="col">Taxes
  <tbody>
    <tr>
      <td>$ 5.00
      <td>$ 4.50
</table>

HTML 换行

可选择性地断开长行。

虽然 HTML 没有列限制建议，但如果显著提高可读性，您可以考虑换行长行。

在换行时，每个续行应缩进以区分包装属性和子元素。行应在项目中一致地换行，理想情况下由自动代码格式化工具强制执行。

<button
  mat-icon-button
  color="primary"
  class="menu-button"
  (click)="openMenu()"
>
  <mat-icon>menu</mat-icon>
</button>

<button mat-icon-button color="primary" class="menu-button"
    (click)="openMenu()">
  <mat-icon>menu</mat-icon>
</button>

<button
    mat-icon-button
    color="primary"
    class="menu-button"
    (click)="openMenu()">
  <mat-icon>menu</mat-icon>
</button>

<button mat-icon-button
        color="primary"
        class="menu-button"
        (click)="openMenu()">
  <mat-icon>menu</mat-icon>
</button>

HTML 引号

在引用属性值时，使用双引号。

在属性值周围使用双引号 ("") 而不是单引号 ('')。

<!-- 不推荐 -->
<a class='maia-button maia-button-secondary'>Sign in</a>

<!-- 推荐 -->
<a class="maia-button maia-button-secondary">Sign in</a>

CSS

CSS 样式规则

CSS 有效性

尽可能使用有效的 CSS。

除非处理 CSS 验证器错误或需要专有语法，否则使用有效的 CSS 代码。

使用诸如 W3C CSS 验证器 之类的工具进行测试。

使用有效的 CSS 是一个可衡量的基准质量属性，它能帮助发现可能无效且可以删除的 CSS 代码，并确保正确使用 CSS。

类名命名

使用有意义或通用的类名。

不要使用表现性或难以理解的名称，始终使用反映所讨论元素目的或其他通用的类名。

应优先使用具体且反映元素目的的名称，因为这些名称最易理解且最不可能发生变化。

通用名称只是对于那些与其兄弟元素没有特别或不同含义的元素的备选方案。它们通常作为“辅助”工具而需要。

使用功能性或通用名称可以减少不必要的文档或模板更改的可能性。

/* 不推荐：无意义 */
.yee-1901 {}

/* 不推荐：表现性 */
.button-green {}
.clear {}

/* 推荐：具体 */
.gallery {}
.login {}
.video {}

/* 推荐：通用 */
.aux {}
.alt {}

类名风格

使用尽可能短但必要时长的类名。

尝试在尽可能简短的情况下传达一个类的用途。

以这种方式使用类名有助于达到可接受的理解性和代码效率水平。

/* 不推荐 */
.navigation {}
.atr {}

/* 推荐 */
.nav {}
.author {}

类名分隔符

在类名中用连字符分隔单词。

为了提高理解性和可扫描性，不要在选择器中用任何字符（包括不使用任何字符）连接单词和缩写，而应使用连字符。

/* 不推荐：未分隔“demo”和“image”单词 */
.demoimage {}

/* 不推荐：使用下划线而不是连字符 */
.error_status {}

/* 推荐 */
.video-id {}
.ads-sample {}

前缀

用特定于应用程序的前缀作为选择器的前缀（可选）。

在大型项目中以及对于嵌入到其他项目或外部网站的代码，使用前缀（作为命名空间）来命名类名。使用短的、唯一的标识符后跟一个连字符。

使用命名空间有助于防止命名冲突，并可以使维护更容易，例如在搜索和替换操作中。

.adw-help {} /* AdWords */
.maia-note {} /* Maia */

类型选择器

避免用类型选择器限定类名。

除非必要（例如使用辅助类），否则不要将元素名称与类结合使用。

避免不必要的祖先选择器对 性能原因有用。

/* 不推荐 */
ul.example {}
div.error {}

/* 推荐 */
.example {}
.error {}

ID 选择器

避免使用 ID 选择器。

ID 属性预计在整个页面中是唯一的，这在页面包含许多由许多不同工程师工作的组件时难以保证。在所有情况下应优先使用类选择器。

/* 不推荐 */
#example {}

/* 推荐 */
.example {}

简写属性

尽可能使用简写属性。

CSS 提供了多种 简写 属性（如 font），应尽可能使用，即使在只明确设置一个值的情况下也是如此。

使用简写属性对代码效率和理解性有用。

/* 不推荐 */
border-top-style: none;
font-family: palatino, georgia, serif;
font-size: 100%;
line-height: 1.6;
padding-bottom: 2em;
padding-left: 1em;
padding-right: 1em;
padding-top: 0;

/* 推荐 */
border-top: 0;
font: 100%/1.6 palatino, georgia, serif;
padding: 0 1em 2em;

0 和单位

在“0”值后省略单位说明，除非需要。

在 0 值后不要使用单位，除非它们是必需的。

flex: 0px; /* 此 flex-basis 组件需要单位。 */
flex: 1 1 0px; /* 没有单位不模糊，但在 IE11 中需要。 */
margin: 0;
padding: 0;

前导 0

始终包含值的前导“0”。

在 -1 到 1 之间的值或长度前放置 0。

font-size: 0.8em;

十六进制表示法

在可能的情况下使用 3 字符的十六进制表示法。

对于允许的颜色值，3 字符的十六进制表示法更短且更简洁。

/* 不推荐 */
color: #eebbcc;

/* 推荐 */
color: #ebc;

重要声明

避免使用 !important 声明。

这些声明打破了CSS的自然级联，使得推理和组合样式变得困难。请使用 选择器特异性 来覆盖属性。

/* 不推荐 */
.example {
  font-weight: bold !important;
}

/* 推荐 */
.example {
  font-weight: bold;
}

Hacks

避免使用用户代理检测以及CSS“hack”——首先尝试其他方法。

通过用户代理检测或特殊的CSS过滤器、解决方案和hack来解决样式差异的诱惑很大。这两种方法都应被视为最后的手段，以实现和维护高效且可管理的代码库。换句话说，给予检测和hack自由通行权将在长期内损害项目，因为项目往往会选择阻力最小的路径。也就是说，允许并使使用检测和hack变得容易，意味着会更频繁地使用检测和hack——而更频繁就是太频繁了。

CSS格式规则

声明顺序

按字母顺序排列声明（可选）。

在项目中一致地排序声明。在没有工具自动化和强制执行一致排序顺序的情况下，考虑将声明按字母顺序排列，以实现一致的代码，并且这种方式易于学习、记忆和手动维护。

为排序目的忽略供应商特定前缀。然而，对于某个CSS属性的多个供应商特定前缀应保持排序（例如，-moz前缀应在-webkit之前）。

background: fuchsia;
border: 1px solid;
-moz-border-radius: 4px;
-webkit-border-radius: 4px;
border-radius: 4px;
color: black;
text-align: center;
text-indent: 2em;

块内容缩进

缩进所有块内容。

缩进所有 块内容， 即规则内的规则以及声明，以反映层次结构并提高理解度。

@media screen, projection {

  html {
    background: #fff;
    color: #444;
  }

}

声明终止符

在每个声明后使用分号。

为了一致性和可扩展性，每个声明都以分号结束。

/* 不推荐 */
.test {
  display: block;
  height: 100px
}

/* 推荐 */
.test {
  display: block;
  height: 100px;
}

属性名终止符

在属性名冒号后使用空格。

为了保持一致性，属性和值之间始终使用单个空格（但属性和冒号之间没有空格）。

/* 不推荐 */
h3 {
  font-weight:bold;
}

/* 推荐 */
h3 {
  font-weight: bold;
}

声明块分隔

在最后一个选择器和声明块之间使用空格。

在最后一个选择器和开始 声明块的左大括号之间始终使用单个空格。

左大括号应与给定规则中的最后一个选择器在同一行上。

/* 不推荐：缺少空格 */
.video{
  margin-top: 1em;
}

/* 不推荐：不必要的换行 */
.video
{
  margin-top: 1em;
}

/* 推荐 */
.video {
  margin-top: 1em;
}

选择器和声明分隔

用新行分隔选择器和声明。

始终为每个选择器和声明开始新行。

/* 不推荐 */
a:focus, a:active {
  position: relative; top: 1px;
}

/* 推荐 */
h1,
h2,
h3 {
  font-weight: normal;
  line-height: 1.2;
}

规则分隔

用新行分隔规则。

规则之间始终留一个空行（两个换行符）。

html {
  background: #fff;
}

body {
  margin: auto;
  width: 50%;
}

CSS引号

对于属性选择器和属性值，使用单引号（''）而不是双引号（""）。

在URI值（url()）中不使用引号。

例外：如果确实需要使用@charset规则，请使用双引号—— 不允许使用单引号。

/* 不推荐 */
@import url("https://www.google.com/css/maia.css");

html {
  font-family: "open sans", arial, sans-serif;
}

/* 推荐 */

临别赠言

保持一致性。

如果你在编辑代码，请花几分钟时间查看周围的代码并确定其风格。如果他们在所有算术运算符周围使用空格，你也应该这样做。如果他们的注释周围有小方框的哈希标记，你的注释也应该有小方框的哈希标记。

制定风格指南的目的是拥有一个共同的编码词汇表，这样人们可以专注于你要表达的内容，而不是你表达的方式。我们在这里提供全局风格规则，以便人们了解词汇表，但本地风格也很重要。如果你添加到文件中的代码与周围现有代码看起来截然不同，当读者去阅读时，会打乱他们的节奏。避免这种情况。
