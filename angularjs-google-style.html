Google Closure 用户的 AngularJS 风格指南

这是主要为 Google 工程师编写的文档的外部版本。它描述了使用 Closure 的 AngularJS 应用程序的推荐风格，正如 Google 内部所使用的那样。更广泛的 AngularJS 社区成员可以根据自己的使用情况自由应用（或不应用）这些建议。

本文档描述了 google3 中 AngularJS 应用程序的风格。此指南补充并扩展了 Google JavaScript 风格指南。

风格注意事项：AngularJS 外部网页上的示例和许多外部应用程序的编写风格自由使用闭包，偏好函数式继承，并且不经常使用 JavaScript 类型。Google 遵循更严格的 JavaScript 风格以支持 JSCompiler 优化和大型代码库 - 请参阅 javascript-style 邮件列表。这不是 Angular 特有的问题，在此风格指南中不再讨论。（但如果你想进一步阅读： Martin Fowler 关于闭包, 更长的描述，附录 A 的 闭包书对继承模式有很好的描述，并解释了为什么它偏好伪经典， JavaScript，好的部分作为对比。）

1 Angular 语言规则

使用 Closure 的 goog.require 和 goog.provide 管理依赖

为您的项目选择一个命名空间，并使用 goog.provide 和 goog.require。

goog.provide('hello.about.AboutCtrl');
goog.provide('hello.versions.Versions');

为什么？ Google BUILD 规则与 closure provide/require 很好地集成。

模块

您的主要应用程序模块应位于根客户端目录中。除定义模块的文件外，不应更改模块。

模块可以定义在与其组件相同的文件中（这对于仅包含一个服务的模块效果很好），也可以在单独的文件中用于连接各个部分。

为什么？ 模块对于任何希望将其作为可重用组件包含的人来说应该是统一的。如果一个模块的含义取决于包含的文件，它就不一致。

模块应使用 Angular 模块的 "name" 属性引用其他模块

例如：

// file submodulea.js:
  goog.provide('my.submoduleA');

  my.submoduleA = angular.module('my.submoduleA', []);
  // ...

// file app.js
  goog.require('my.submoduleA');

  是：my.application.module = angular.module('hello', [my.submoduleA.name]);
  
      否：my.application.module = angular.module('hello', ['my.submoduleA']);

为什么？ 使用 my.submoduleA 的属性可以防止 Closure 预提交失败，抱怨文件被要求但从未使用。使用 .name 属性避免重复字符串。

使用通用的 externs 文件

这最大限度地允许 JS 编译器在存在 Angular 提供的外部类型的情况下强制执行类型安全性，并且意味着您不必担心 Angular 变量以令人困惑的方式被混淆。

注意：对于 Google 之外的读者：当前的 externs 文件位于 Google 内部目录中，但可以在 github 上找到示例 这里。

JSCompiler 标志

提醒：根据 JS 风格指南，面向客户的代码必须编译。

推荐：使用 JSCompiler（默认情况下与 js_binary 一起使用的 closure 编译器）和来自 //javascript/angular/build_defs/build_defs 的 ANGULAR_COMPILER_FLAGS_FULL 作为您的基本标志。

注意 - 如果您使用 @export 标记方法，您需要添加编译器标志

"--generate_exports",

如果您使用 @export 标记属性，您需要添加标志：

"--generate_exports",
"--remove_unused_prototype_props_in_externs=false",
"--export_local_property_definitions",

控制器和作用域

控制器是类。方法应定义在 MyCtrl.prototype 上。

Google Angular 应用程序应使用 'controller as' 风格将控制器导出到作用域上。这在 Angular 1.2 中完全实现，并且可以在 Angular 1.2 之前的版本中模拟。

在 Angular 1.2 之前，看起来像这样：

/**
 * 主页控制器。
 *
 * @param {!angular.Scope} $scope
 * @constructor
 * @ngInject
 * @export
 */
hello.mainpage.HomeCtrl = function($scope) {
  /** @export */
  $scope.homeCtrl = this; // 这是一个桥梁，直到 Angular 1.2 的 controller-as

  /**
   * @type {string}
   * @export
   */
  this.myColor = 'blue';
};


/**
 * @param {number} a
 * @param {number} b
 * @export
 */
hello.mainpage.HomeCtrl.prototype.add = function(a, b) {
  return a + b;
};

以及模板：

<div ng-controller="hello.mainpage.HomeCtrl"/>
  <span ng-class="homeCtrl.myColor">我有颜色！</span>
  <span>{{homeCtrl.add(5, 6)}}</span>
</div>

在 Angular 1.2 之后，看起来像这样：

/**
 * 主页控制器。
 *
 * @constructor
 * @ngInject
 * @export
 */
hello.mainpage.HomeCtrl = function() {
  /**
   * @type {string}
   * @export
   */
  this.myColor = 'blue';
};


/**
 * @param {number} a
 * @param {number} b
 * @export
 */
hello.mainpage.HomeCtrl.prototype.add = function(a, b) {
  return a + b;
};

如果您使用属性重命名进行编译，请使用 @export 注释暴露属性和方法。记得也 @export 构造函数。

以及在模板中：

<div ng-controller="hello.mainpage.HomeCtrl as homeCtrl"/>
  <span ng-class="homeCtrl.myColor">我有颜色！</span>
  <span>{{homeCtrl.add(5, 6)}}</span>
</div>

为什么？ 将方法和属性直接放到控制器上，而不是构建一个作用域对象，更符合 Google Closure 类的风格。此外，使用 'controller as' 使得在多个控制器应用于一个元素时，很明显您正在访问哪个控制器。由于绑定中总是有一个 '.'，您不必担心原型继承掩盖基本类型的问题。

指令

所有 DOM 操作都应在指令内完成。指令应保持小巧并使用组合。定义指令的文件应 goog.provide 一个返回指令定义对象的静态函数。

goog.provide('hello.pane.paneDirective');

/**
 * 描述和使用
 * @return {angular.Directive} 指令定义对象。
 */
hello.pane.paneDirective = function() {
  // ...
};

例外：对于与视图其余部分断开连接的 DOM 元素（例如对话框或键盘快捷键），可以在服务中进行 DOM 操作。

服务

在模块上使用 module.service 注册的服务是类。 使用 module.service 而不是 module.provider 或 module.factory，除非您需要进行超出仅创建类新实例的初始化。

/**
 * @param {!angular.$http} $http Angular http 服务。
 * @constructor
 */
hello.request.Request = function($http) {
  /** @type {!angular.$http} */
  this.http_ = $http;
};

hello.request.Request.prototype.get = function() {/*...*/};

在模块中：

module.service('request', hello.request.Request);

2 Angular 风格规则

保留 $ 用于 Angular 属性和服务

不要使用 $ 来前置您自己的对象属性和服务标识符。请将这种命名风格视为 AngularJS 和 jQuery 保留的。

是：

  $scope.myModel = { value: 'foo' }
  myModule.service('myService', function() { /*...*/ });
  var MyCtrl = function($http) {this.http_ = $http;};

否：

  $scope.$myModel = { value: 'foo' } // 错误
  $scope.myModel = { $value: 'foo' } // 错误
  myModule.service('$myService', function() { ... }); // 错误
  var MyCtrl = function($http) {this.$http_ = $http;}; // 错误

为什么？ 区分 Angular / jQuery 内置和您自己添加的东西是有用的。此外，$ 在 JS 风格指南中不是变量名的可接受字符。

自定义元素

对于自定义元素（例如 <ng-include src="template"></ng-include>），IE8 需要特殊支持（类似 html5shiv 的 hack）来启用 css 样式。请注意此限制，适用于针对旧版本 IE 的应用程序。

使用 @ngInject 进行简单的依赖注入编译

这消除了添加 myCtrl['$inject'] = ... 的需要，以防止压缩混淆 Angular 的依赖注入。

使用方法：

/**
 * 我的控制器。
 * @param {!angular.$http} $http
 * @param {!my.app.myService} myService
 * @constructor
 * @export
 * @ngInject
 */
my.app.MyCtrl = function($http, myService) {
  //...
};
